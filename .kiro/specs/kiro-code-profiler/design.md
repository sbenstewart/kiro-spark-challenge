# Design Document: Kiro Code Profiler

## Overview

The Kiro Code Profiler is a Kiro IDE extension that instruments and measures resource usage of JavaScript, TypeScript, and Python code. It provides execution profiling, real-time monitoring, AI-assisted optimization suggestions, and session history — all surfaced within the IDE without requiring external tools.

The extension is structured as a VS Code–compatible extension (Kiro is built on VS Code) with a TypeScript backend orchestrating child-process execution, metric collection, and persistence, and a Webview-based dashboard for visualization.

Key design decisions:
- **Out-of-process execution**: Code runs in a child process to isolate metrics and prevent IDE instability.
- **Polling-based metric collection**: OS-level APIs (via `pidusage`, `systeminformation`, and runtime hooks) are polled at configurable intervals rather than using kernel tracing, keeping the implementation cross-platform.
- **Local persistence**: Sessions are stored as newline-delimited JSON files per workspace, avoiding any external database dependency.
- **AI suggestions via Kiro AI**: Optimization suggestions are generated by calling the Kiro AI API with the profiling context, keeping the optimizer stateless.

### Data Flow

1. Developer triggers profiling via command palette or toolbar button.
2. `Extension Entry Point` reads configuration and invokes `Execution Runner`.
3. `Execution Runner` spawns a child process and starts `Metrics Collector` polling.
4. On process exit, `Metrics Collector` finalizes metrics; `Energy Estimator` computes energy.
5. `Session Persister` writes the session to disk.
6. `Webview Dashboard` receives the session via message passing and renders it.
7. If thresholds are exceeded, `Optimizer` is invoked and suggestions are displayed.

## Components and Interfaces

### Execution Runner

Responsible for spawning the user's code in an isolated child process.

```typescript
interface ExecutionRunner {
  run(request: RunRequest): Promise<ExecutionResult>;
}

interface RunRequest {
  filePath: string;
  language: 'javascript' | 'typescript' | 'python';
  runtimePath?: string; // custom runtime override
  selectedCode?: string; // if set, run this snippet instead of the full file
}

interface ExecutionResult {
  exitCode: number;
  stdout: string;
  stderr: string;
  startTime: number; // epoch ms
  endTime: number;   // epoch ms
}
```

### Metrics Collector

Polls OS-level metrics for a given PID at a configurable interval.

```typescript
interface MetricsCollector {
  start(pid: number, intervalMs: number): void;
  stop(): MetricsSummary;
  getSamples(): MetricSample[];
}

interface MetricSample {
  timestamp: number; // epoch ms
  ramMb: number;
  cpuPercent: number;
  diskReadBytes: number;
  diskWriteBytes: number;
  networkBytesSent: number;
  networkBytesReceived: number;
  fsOpen: number;
  fsRead: number;
  fsWrite: number;
  fsClose: number;
}

interface MetricsSummary {
  peakRamMb: number;
  avgRamMb: number;
  totalDiskReadBytes: number;
  totalDiskWriteBytes: number;
  avgCpuPercent: number;
  totalNetworkBytesSent: number;
  totalNetworkBytesReceived: number;
  totalFsOpen: number;
  totalFsRead: number;
  totalFsWrite: number;
  totalFsClose: number;
  gcEvents?: number;
  gcPauseTotalMs?: number;
  executionTimeMs: number;
  energyMwh: number;
  samples: MetricSample[];
}
```

### Energy Estimator

Estimates energy consumption from CPU utilization and execution time.

```typescript
interface EnergyEstimator {
  estimate(avgCpuPercent: number, executionTimeMs: number, tdpWatts?: number): number; // returns mWh
}
```

Default TDP is read from system info (via `systeminformation`) or falls back to a conservative 15W estimate.

### Optimizer

Calls Kiro AI to generate ranked optimization suggestions.

```typescript
interface Optimizer {
  suggest(session: ProfileSession, sourceCode: string): Promise<OptimizationSuggestion[]>;
}

interface OptimizationSuggestion {
  id: string;
  title: string;
  explanation: string;       // plain-language reason
  estimatedImpact: number;   // 0–1 normalized score
  affectedMetric: 'ram' | 'cpu' | 'energy' | 'disk' | 'network';
  diff: string;              // unified diff format
}
```

Suggestions are sorted by `estimatedImpact` descending before being returned.

### Monitor

Attaches to a running process (by PID or by launching one) and streams samples.

```typescript
interface Monitor {
  attach(pid: number, config: MonitorConfig): void;
  launch(request: RunRequest, config: MonitorConfig): Promise<number>; // returns PID
  stop(): Promise<ProfileSession>;
  on(event: 'sample', handler: (sample: MetricSample) => void): void;
  on(event: 'alert', handler: (alert: MetricAlert) => void): void;
}

interface MonitorConfig {
  sampleIntervalMs: number;
  ramAlertThresholdMb: number;
  cpuAlertThresholdPercent: number;
}

interface MetricAlert {
  type: 'ram' | 'cpu';
  value: number;
  threshold: number;
  timestamp: number;
}
```

### Session Persister

Reads and writes sessions to disk.

```typescript
interface SessionPersister {
  save(session: ProfileSession): Promise<void>;
  list(workspacePath: string): Promise<SessionSummary[]>;
  load(sessionId: string): Promise<ProfileSession>;
  clear(workspacePath: string): Promise<void>;
  purgeExpired(workspacePath: string, retentionDays: number): Promise<void>;
}
```

Sessions are stored at `.kiro/profiler/sessions/{sessionId}.json` within the workspace.

### Configuration Manager

Wraps VS Code / Kiro workspace configuration.

```typescript
interface ProfilerConfig {
  ramAlertThresholdMb: number;       // default: 512
  cpuAlertThresholdPercent: number;  // default: 80
  sampleIntervalMs: number;          // default: 1000, min: 100
  runtimePaths: {
    node?: string;
    python?: string;
  };
}
```

## Data Models

### ProfileSession

The canonical record for a completed profiling or monitoring session.

```typescript
interface ProfileSession {
  id: string;                    // UUID v4
  workspacePath: string;
  filePath: string;
  language: 'javascript' | 'typescript' | 'python';
  sessionType: 'profile' | 'monitor';
  startTime: number;             // epoch ms
  endTime: number;               // epoch ms
  exitCode: number;
  stdout: string;
  stderr: string;
  metrics: MetricsSummary;
  isBaseline: boolean;
  optimizationSuggestions: OptimizationSuggestion[];
  linkedPreSessionId?: string;   // for post-optimization comparison
}
```

### SessionSummary

Lightweight record for the history list (avoids loading full sample arrays).

```typescript
interface SessionSummary {
  id: string;
  filePath: string;
  sessionType: 'profile' | 'monitor';
  startTime: number;
  endTime: number;
  peakRamMb: number;
  avgCpuPercent: number;
  executionTimeMs: number;
  isBaseline: boolean;
}
```

### Baseline Comparison

```typescript
interface BaselineComparison {
  baselineSessionId: string;
  currentSessionId: string;
  deltas: {
    ramMb: number;        // percentage delta
    cpuPercent: number;
    energyMwh: number;
    executionTimeMs: number;
    diskReadBytes: number;
    diskWriteBytes: number;
  };
}
```

Delta formula: `(current - baseline) / baseline * 100`

## Correctness Properties

*A property is a characteristic or behavior that should hold true across all valid executions of a system — essentially, a formal statement about what the system should do. Properties serve as the bridge between human-readable specifications and machine-verifiable correctness guarantees.*

### Property 1: Metrics summary completeness

*For any* completed profiling session, the returned `MetricsSummary` object SHALL contain all required numeric fields: `peakRamMb`, `avgRamMb`, `totalDiskReadBytes`, `totalDiskWriteBytes`, `avgCpuPercent`, `executionTimeMs`, and `energyMwh`, each with a finite non-negative numeric value.

**Validates: Requirements 1.2, 1.3, 1.4, 1.5, 3.1**

---

### Property 2: Error exit captures partial metrics

*For any* process that exits with a non-zero exit code, the `ExecutionResult` SHALL contain a non-empty `stderr` string, and the associated `MetricsSummary` SHALL contain whatever samples were collected before exit (i.e., `samples.length >= 0` and `executionTimeMs >= 0`).

**Validates: Requirements 1.6**

---

### Property 3: Time-series sampling interval

*For any* profiling session with at least two samples, the difference between consecutive sample timestamps SHALL be no greater than 100ms.

**Validates: Requirements 2.2, 2.3**

---

### Property 4: Baseline delta correctness

*For any* pair of sessions (baseline and current), the computed percentage delta for each metric SHALL equal `(current - baseline) / baseline * 100`, rounded to two decimal places, for all metrics where the baseline value is non-zero.

**Validates: Requirements 2.4, 5.4**

---

### Property 5: Suggestions have explanations and are ranked

*For any* list of optimization suggestions returned by the Optimizer, every suggestion SHALL have a non-empty `explanation` string, and the list SHALL be sorted by `estimatedImpact` in descending order (i.e., `suggestions[i].estimatedImpact >= suggestions[i+1].estimatedImpact` for all valid `i`).

**Validates: Requirements 4.2, 4.6**

---

### Property 6: Suggestion rejection is a no-op

*For any* original source code string and any optimization suggestion, applying the suggestion as a preview and then rejecting it SHALL result in the editor buffer content being byte-for-byte identical to the original source code.

**Validates: Requirements 4.5**

---

### Property 7: Alert emission on threshold breach

*For any* metric sample where `ramMb > config.ramAlertThresholdMb` OR `cpuPercent > config.cpuAlertThresholdPercent`, the Monitor SHALL emit at least one `alert` event with the correct `type`, `value`, and `threshold` fields.

**Validates: Requirements 6.4, 6.5**

---

### Property 8: Monitor sampling interval

*For any* active monitoring session with at least two samples, the elapsed time between consecutive samples SHALL not exceed `config.sampleIntervalMs + 50ms` (allowing 50ms scheduling jitter).

**Validates: Requirements 6.1**

---

### Property 9: Session persistence round-trip

*For any* completed `ProfileSession` object, serializing it to disk via `SessionPersister.save()` and then loading it back via `SessionPersister.load(session.id)` SHALL produce an object that is deeply equal to the original.

**Validates: Requirements 7.1, 7.3**

---

### Property 10: Session history ordering

*For any* workspace with multiple persisted sessions, `SessionPersister.list()` SHALL return sessions ordered by `startTime` descending (most recent first).

**Validates: Requirements 7.2**

---

### Property 11: Session retention policy

*For any* session with `startTime` within the last 30 days, `purgeExpired()` SHALL NOT remove it. For any session with `startTime` older than 30 days, it SHALL be eligible for removal.

**Validates: Requirements 7.4**

---

### Property 12: Clear removes all sessions

*For any* workspace with any number of persisted sessions, after calling `SessionPersister.clear()`, `SessionPersister.list()` SHALL return an empty array.

**Validates: Requirements 7.5**

---

### Property 13: Configuration validation

*For any* sample interval value less than 100ms, the Configuration Manager SHALL reject it (throw or return a validation error). For any value >= 100ms, it SHALL be accepted.

**Validates: Requirements 8.4**

---

### Property 14: Threshold suggestions triggered

*For any* completed session where at least one metric value exceeds its configured threshold, the Optimizer SHALL return at least one suggestion.

**Validates: Requirements 4.1**

---

## Error Handling

| Scenario | Behavior |
|---|---|
| Child process fails to spawn (bad runtime path) | Surface error in Dashboard with actionable message; no session persisted |
| Child process times out (configurable, default 5 min) | Kill process, persist partial session with `exitCode: -1` |
| Metrics Collector loses PID (process already exited) | Stop polling, finalize with collected samples |
| AI API unavailable | Show inline error in suggestions panel; session still persisted |
| Disk write failure on session persist | Log error to extension output channel; notify user via status bar |
| PID not found when attaching Monitor | Return error immediately; do not start monitoring session |
| Configuration value out of range | Clamp to valid range and log warning; never silently use invalid value |

## Testing Strategy

### Unit Tests

Focus on pure logic components:
- `EnergyEstimator.estimate()` — verify formula correctness with known inputs
- `BaselineComparison` delta calculation — verify percentage formula
- `SessionPersister` serialization/deserialization — verify round-trip fidelity
- `Optimizer` suggestion ranking — verify sort order
- Configuration validation — verify boundary conditions (e.g., sampleIntervalMs < 100)
- `MetricsSummary` aggregation from raw samples — verify peak/avg calculations

### Property-Based Tests

Use **fast-check** (TypeScript property-based testing library) with a minimum of 100 iterations per property.

Each test is tagged with the corresponding design property:

```
// Feature: kiro-code-profiler, Property 1: Metrics summary completeness
// Feature: kiro-code-profiler, Property 4: Baseline delta correctness
// ...
```

Properties to implement:
- **Property 1**: Generate arbitrary `MetricSample[]` arrays, aggregate into `MetricsSummary`, assert all required fields are finite non-negative numbers.
- **Property 2**: Generate arbitrary non-zero exit codes, assert result contains stderr and non-negative executionTimeMs.
- **Property 3**: Generate arbitrary sample arrays, assert consecutive timestamp differences ≤ 100ms.
- **Property 4**: Generate arbitrary pairs of numeric metric values, assert delta formula holds.
- **Property 5**: Generate arbitrary suggestion lists, assert all have non-empty explanations and are sorted by impact.
- **Property 6**: Generate arbitrary source code strings and diffs, apply then reject, assert content unchanged.
- **Property 7**: Generate arbitrary samples with values above/below thresholds, assert alert emission behavior.
- **Property 8**: Generate arbitrary monitoring sessions, assert sample interval constraint.
- **Property 9**: Generate arbitrary `ProfileSession` objects, save and load, assert deep equality.
- **Property 10**: Generate arbitrary session lists with varying timestamps, assert ordering.
- **Property 11**: Generate sessions with timestamps relative to now, assert retention policy.
- **Property 12**: Generate arbitrary session sets, clear, assert empty list.
- **Property 13**: Generate arbitrary interval values, assert acceptance/rejection at 100ms boundary.
- **Property 14**: Generate arbitrary metrics with at least one threshold breach, assert at least one suggestion returned.

### Integration Tests

- End-to-end profiling of a minimal JS/TS/Python script — verify session is persisted and metrics are non-zero.
- Dashboard receives session data within 500ms of process exit.
- Monitor attaches to a running Node.js process by PID and collects at least 3 samples.

### Smoke Tests

- Extension activates without errors in a fresh Kiro workspace.
- Configuration schema is registered and defaults are correct (512 MB RAM, 80% CPU, 1000ms interval).
